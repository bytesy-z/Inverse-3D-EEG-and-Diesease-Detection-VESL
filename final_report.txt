Abstract
EEG source localization—inferring brain activity from scalp potentials—is a
fundamentally ill-posed inverse problem critical to pre-surgical evaluation of
drug-resistant epilepsy. This project presents PhysDeepSIF, a physics-informed deep
learning framework that reconstructs 76-region cortical source activity from 19-channel
scalp EEG (10-20 montage, linked-ear reference). The system integrates a hybrid neural
network (spatial MLP + BiLSTM) trained on synthetic data generated via the TVB
Epileptor model, with a two-phase inference pipeline: (A) instantaneous biomarker
detection using power-based scoring, and (B) patient-specific biophysical fitting
via CMA-ES optimization of the Epileptor excitability parameter x₀. Evaluated
on held-out synthetic test data, the network achieves a Distance Localization Error
(DLE) of 31.08 mm (centroid-based), outperforming all classical inverse methods
by 18–25 mm (eLORETA: 53.54 mm, MNE: 49.55 mm, sLORETA: 56.52 mm). Source
localization AUC is 0.697 vs. random baseline 0.502. A concordance metric between
Phase A and Phase B produces tiered confidence levels (HIGH/MODERATE/LOW). The
system is deployed as a full-stack web application (FastAPI backend, Next.js frontend)
with real-time WebSocket progress, interactive 3D brain heatmaps, and explainable
AI (XAI) occlusion analysis. The framework provides a biophysically grounded,
data-driven approach to epileptogenic zone localization using only non-invasive
scalp EEG.



1  Background

1.1  Problem Statement

Approximately 30% of epilepsy patients are drug-resistant and may benefit from
resective surgery. Successful surgery depends on accurately localizing the
epileptogenic zone (EZ)—the cortical region responsible for generating seizures.
Current clinical practice relies on invasive intracranial EEG (iEEG) or expensive
multimodal imaging (MRI, PET, SPECT, MEG), which are not universally available,
carry surgical risk, and incur substantial cost.

Scalp EEG is universally accessible, non-invasive, and inexpensive. However, the
EEG inverse problem—estimating brain source activity from scalp potentials—is
severely underdetermined: infinitely many source configurations can produce identical
scalp measurements. Classical distributed inverse methods (e.g., eLORETA, sLORETA,
MNE, dSPM) rely on explicit regularization (minimum-norm, smoothness, or anatomical
priors) but suffer from low spatial resolution (typically >30 mm DLE) and cannot
incorporate biophysical dynamics of seizure generation.

Deep learning approaches to EEG source imaging exist (DeepSIF, 2022) but operate at
vertex-level resolution (~994 vertices) requiring dense electrode coverage and do
not provide patient-specific biophysical validation. No existing system combines
fast deep-learning-based source inversion with mechanistic model fitting for
clinical-grade epileptogenicity mapping using only routine 19-channel EEG.

1.2  Literature Review

EEG source imaging has been studied for decades. Classical approaches include:

Minimum Norm Estimate (MNE): The simplest distributed inverse, selecting the
source configuration with minimum L2 norm that explains the data (Hämäläinen &
Ilmoniemi, 1994). Suffers from bias toward superficial sources.

sLORETA / eLORETA: Standardized low-resolution electromagnetic tomography
(Pascual-Marqui, 2002, 2011) uses weighted minimum-norm with zero localization
error under ideal conditions. eLORETA is the most widely cited benchmark
for EEG source localization; our reproduced baseline achieves 53.54 mm
centroid DLE on the synthetic3 test set.

Beamformers (LCMV, DICS): Spatial filtering approaches that suppress
contributions from other regions (Van Veen et al., 1997). Sensitive to
correlated sources and requires accurate noise covariance.

Deep Learning for Source Imaging: DeepSIF (Sun et al., 2022, PNAS) introduced
a hybrid MLP-LSTM architecture trained on synthetic spike data (Wendling model)
at vertex-level resolution (~994 vertices). Achieved superior DLE to classical
methods but required dense electrode coverage and lacked biophysical constraints
beyond forward consistency.

The Virtual Brain (TVB): A neuroinformatics platform for large-scale neural
mass simulations (Sanz-Leon et al., 2013). The Epileptor model (Jirsa et al.,
2014) captures seizure dynamics through a 6-D ODE system per brain region.
Vattikonda et al. (2021) and the Virtual Epileptic Patient framework (Jirsa et
al., 2017) demonstrated that fitting the x₀ excitability parameter can identify
epileptogenic regions, but typically requires hours per patient using stochastic
optimization.

Physics-Informed Neural Networks (PINNs): Introduced by Raissi et al. (2019),
PINNs incorporate physical laws as soft constraints in the loss function. This
project extends the PINN paradigm to EEG source imaging by embedding the
leadfield forward model and biophysical regularization terms.

1.3  Proposed Solution

We propose PhysDeepSIF, a physics-informed deep source imaging framework with
patient-specific biophysical validation, comprising:

1. A hybrid deep neural network (spatial MLP + BiLSTM) that maps 19-channel EEG
   to 76-region cortical source activity. Trained entirely on synthetic data
   generated via the TVB Epileptor model, eliminating the need for ground-truth
   source-level labels from real patients.

2. A composite physics-informed loss function: source reconstruction MSE +
   amplitude regularization + temporal smoothness. The loss incorporates
   structural connectivity and leadfield matrices as frozen physical buffers.

3. A two-phase inference pipeline:
   Phase A — Instantaneous biomarker detection using AC variance scoring,
   z-score normalization, and sigmoid-based Epileptogenicity Index (EI).
   Phase B — CMA-ES evolutionary optimization fitting the Epileptor x₀
   excitability parameter to minimize PSD mismatch between simulated and
   real EEG at the scalp level.

4. Concordance analysis computing overlap between Phase A (data-driven) and
   Phase B (model-driven) top-10 regions, providing confidence tiers (HIGH,
   MODERATE, LOW).

5. Explainable AI via occlusion-based channel importance and temporal attribution.

6. A full-stack web application (FastAPI + Next.js) enabling file upload,
   interactive 3D brain visualization, real-time optimization progress, and
   accessible results export.

1.4  Project Scope

The project encompasses:
- Construction of a 76-region source space using the Desikan-Killiany atlas
  and a 19×76 BEM leadfield matrix with linked-ear re-referencing.
- Generation of synthetic EEG training datasets via TVB Epileptor simulations
  (23,000 training, 2,900 validation, 2,900 test simulations; 5 × 2 s windows
  per simulation at 200 Hz).
- Design, training, and evaluation of the PhysDeepSIF inverse network.
- Implementation of Phase A biomarker detection and Phase B CMA-ES biophysical
  fitting.
- Development of a web-based analysis dashboard supporting EDF/MAT/NPY uploads
  and synthetic test sample selection.
- Validation against classical eLORETA baseline on DLE, AUC, top-K recall, and
  hemisphere classification metrics.
- Docker-based deployment configuration.

The project does NOT include:
- Vertex-level source reconstruction (region-level only).
- Patient-specific MRI processing or individualized head models.
- Real-time clinical deployment or regulatory approval.
- Training or validation on real clinical EEG data (all training is synthetic).

  out of scope.

3  Software Design Specifications

3.1  Architectural Style: Layered Architecture

3.1.1  Justification

The system adopts a layered architecture for the following reasons:

- Separation of Concerns: Each layer addresses a distinct responsibility:
  data management, computation, API serving, and user interaction. This
  isolates the computationally intensive inference and optimization modules
  from the lightweight web-serving layer.

- Independent Evolvability: The network architecture (Phase 2), CMA-ES optimizer
  (Phase 4), and visualization (frontend) can be modified independently. A new
  neural mass model can be swapped into the synthetic data pipeline without
  affecting the inference API.

- Technology Heterogeneity: The backend uses Python (PyTorch, MNE, TVB)
  while the frontend uses TypeScript/Next.js. The API layer provides a clean
  HTTP/WebSocket interface between these technology stacks.

- Testability: Each layer can be tested in isolation. Unit tests target
  individual modules; integration tests exercise the API layer with mocked
  inference; system tests validate the full stack.

3.1.2  Detailed Mapping of Modules to Layers

Presentation Layer:
  - Frontend Dashboard (Next.js, React 19, TypeScript)
  - Components: brain-visualization.tsx, xai-panel.tsx, concordance-badge.tsx,
    processing-window.tsx, eeg-waveform-viewer.tsx, error-boundary.tsx
  - WebSocket hook: use-websocket.ts

Application Layer (API Gateway):
  - FastAPI server (backend/server.py: 7 HTTP + 1 WebSocket endpoints)
  - Request validation (Pydantic models)
  - Rate limiting middleware
  - CORS middleware
  - Job tracking with thread-safe lock
  - Structured JSON logging with rotation

Business Logic Layer (Core Pipeline):
  - Phase 1: Forward Modeling — epileptor_simulator.py, synthetic_dataset.py,
              leadfield_builder.py, parameter_sampler.py, source_space.py
  - Phase 2: Network — physdeepsif.py, loss_functions.py, trainer.py, metrics.py
  - Phase 3: Inference — preprocessing pipeline, sliding-window segmentation
  - Phase 4: Inversion — cmaes_optimizer.py, objective_function.py,
              epileptogenicity_index.py, concordance.py
  - Phase 5: Validation — generate_figures.py (6 publication figures)
  - XAI: eeg_occlusion.py

Data Layer:
  - HDF5 synthetic datasets (train/val/test)
  - Leadfield, connectivity, region labels/centers/tract lengths (NumPy/JSON)
  - Model checkpoints + normalization statistics (PyTorch/JSON)
  - Frontend results artifacts (HTML, CSV)

4  System Design

4.1  Design Methodology

The project follows a hybrid design approach: procedural programming for the
computational pipeline (data generation, training, inference, validation) and
component-based architecture for the web application.

4.1.1  Justification

- Procedural programming suits the sequential, data-flow-oriented nature of the
  scientific pipeline: simulation → EEG generation → network training → inference
  → biophysical fitting → heatmap. Each phase consumes well-defined inputs and
  produces well-defined outputs, with minimal shared mutable state.

- Python's flexibility supports both procedural scripting and object-oriented
  design. The pipeline modules (e.g., physdeepsif.py, cmaes_optimizer.py) use
  classes for stateful components (model, optimizer) and functions for stateless
  transformations (metrics, preprocessing).

- The frontend uses React's component-based architecture with functional
  components and hooks, appropriate for the event-driven, stateful nature of
  user interaction (file upload, progress tracking, result rendering).

- Lower overhead compared to full object-oriented frameworks for a system with
  limited inter-component interaction complexity. The pipeline does not require
  inheritance hierarchies or extensive object reuse.

4.2.2  Data Management

Data Input:
- Synthetic data generation: Epileptor parameters (x₀ ∈ [−2.2, −1.2], coupling
  ∈ [0.5, 3.0], noise ∈ [1e−4, 5e−3]) sampled from uniform distributions.
  Channel gain values derived from leadfield projection of simulated source
  activity.
- Inference: 19-channel EEG recordings (EDF, MAT, NPY) or synthetic test
  samples selected by index from HDF5 dataset.

Data Processing:
- Synthetic pipeline: TVB simulation (dt=0.1 ms, HeunStochastic integrator) →
  FIR decimation (×100) to 200 Hz → leadfield projection → noise injection
  (white Gaussian 5–30 dB SNR + 1/f^α colored 10–30% amplitude) → skull
  attenuation (40 Hz LP filter) → STFT spectral shaping → window segmentation
  (400 samples, 50% overlap) → HDF5 storage.
- Inference: Global z-score normalization (no per-channel de-meaning) →
  PhysDeepSIF forward pass → source activity estimates → biomarker scoring
  or CMA-ES fitting.

Data Output:
- Source activity estimates: 76 × 400 array per 2-second window.
- Phase A: top-10 regions with anatomical names and EI scores ∈ [0, 1].
- Phase B: 76 fitted x₀ values, biophysical EI scores, concordance tier
  (HIGH/MODERATE/LOW).
- Visualization: Interactive Plotly 3D brain heatmap, XAI channel importance
  plot, temporal attribution heatmap, animation frames for multi-window EEG.

4.3.2  Data Storage Format

Synthetic Dataset (HDF5):

Each sample in the HDF5 file contains:

  Group: /sample_XXXXX/
    Dataset: eeg              shape=(19, 400)    dtype=float32
             Source EEG (19 channels, 400 time points at 200 Hz)
    Dataset: source_activity  shape=(76, 400)    dtype=float32
             Ground-truth source activity (76 regions, 400 time points)
    Dataset: epileptogenic_mask  shape=(76,)     dtype=bool
             Binary mask: True for regions set as epileptogenic
    Dataset: x0_vector        shape=(76,)        dtype=float32
             Excitability parameter for each region
    Attribute: snr_db         scalar float32
             Signal-to-noise ratio in dB (5–30)
    Attribute: global_coupling  scalar float32
             Global coupling strength (0.5–3.0)

Model Checkpoint (PyTorch):

  {
    'model_state_dict': OrderedDict { ... },   # ~419K parameters
    'optimizer_state_dict': { ... },
    'epoch': 47,
    'val_loss': 1.34,
    'config': { ... }                           # Training hyperparameters
  }

Normalization Statistics (JSON):

  {
    "eeg_mean": float,  # Global mean across training set
    "eeg_std":  float,  # Global standard deviation (DC+AC combined)
    "src_mean": float,
    "src_std":  float
  }

5  System Testing

5.1  Application Overview

The system under test is a full-stack web application comprising:
- FastAPI backend (port 8000) serving 7 HTTP endpoints and 1 WebSocket endpoint.
- Next.js frontend (port 3000) providing an interactive data dashboard.
- PhysDeepSIF neural network inference with physics-informed loss.
- CMA-ES evolutionary optimizer for biophysical parameter fitting.
- XAI occlusion analysis module.
- Synthetic data generation and model training pipelines.

5.2  Scope

This test plan covers:
- All API endpoints (health, analyze, results, test-samples).
- Model inference correctness (output shapes, value ranges, finiteness).
- Biomarker detection accuracy (top-10 ordering, EI thresholds).
- CMA-ES optimizer convergence and concordance computation.
- XAI occlusion attribution validity.
- WebSocket progress streaming.
- File upload validation (format, size, channel count).
- Error handling (NaN inputs, oversized files, missing dependencies).
- Security (CORS, rate limiting, input sanitization).

Testing is performed at five layers:
- Unit Tests: Isolated functions and classes.
- Functional Tests: Component-to-component data flow.
- Integration Tests: API endpoints with model loading.
- System Tests: Full pipeline determinism, memory, throughput, device fallback.
- Regression Tests: Bug-triggered scenarios ensuring fixes persist.

5.3  Functional Testing

5.3.1  Unit Testing (18 test files)

Model Components (tests/unit/):
- test_spatial_module.py: Verify 5-layer MLP produces shape (batch, 76, 400)
  from input (batch, 19, 400). Check BatchNorm and skip connections propagate.
- test_temporal_module.py: Verify 2-layer BiLSTM output shape and residual
  skip behavior. Test with varying sequence lengths.
- test_loss_functions.py: Validate each loss component (L_source, L_epi,
  L_physics) individually and in composite. Confirm gradient flow.
- test_dle_metric.py: Distance Localization Error calculation. Test with
  perfect prediction (DLE=0), far-away prediction (DLE > 50 mm), and random.
- test_preprocessing.py: Z-score normalization roundtrip, sliding-window
  segmentation edge cases, EDF channel extraction.
- test_metrics.py: AUC, top-K recall, temporal correlation, spatial dispersion
  on controlled toy data with known expected values.
- test_region_names.py: Verify all 76 DK region codes map to correct anatomical
  names and hemisphere assignments.
- test_ei_computation.py: Sigmoid activation, z-score normalization of AC
  variance, boundary cases (all-zero input, extreme values).
- test_concordance.py: Overlap computation between two top-10 lists. HIGH/
  MODERATE/LOW tier thresholds.

XAI Module (tests/test_xai.py):
- verify_occlusion_shapes: Occlusion output matches expected dimensions.
- verify_channel_importance_sum: Importance scores normalise to 1.0.
- verify_masking_effect: Occluded regions reduce source amplitude.
- verify_edge_cases: Handle single-channel, single-time-point inputs.

Data Pipeline (functional tests):
- test_dataset_generation_roundtrip.py: Generate small synthetic dataset
  (10 sims), verify HDF5 structure, read back and check shapes/values.
- test_training_one_epoch.py: Run one training epoch on synthetic data,
  verify loss decreases, checkpoint saves correctly.
- test_checkpoint_roundtrip.py: Save model, load model, verify output
  identical for same input.

5.4  Integration Testing

API Endpoints (tests/test_api.py):
- /api/health: Verify 200 response, JSON structure, model_loaded=true,
  correct num_regions, device field present.
- /api/test-samples: Verify returns list of available sample indices.
- /api/analyze (synthetic sample): POST with sample_idx=10, mode=biomarkers.
  Verify 200 response, EI scores ∈ [0,1], top-10 regions present.
- /api/analyze (synthetic sample, source_localization mode): Verify CMA-ES
  runs, returns fitted x₀, biophysical EI, concordance tier.
- /api/analyze (EDF upload): Upload test_demo.edf, verify preprocessing,
  inference, and results generation.
- /api/results/{job_id}/{path}: Verify HTML heatmap, CSV tables served
  correctly with appropriate Content-Type headers.

Error Handling (tests/test_api_errors.py):
- Missing file: POST without file or sample_idx → 422.
- Invalid sample index: sample_idx=999999 → 404.
- Corrupt file: Upload non-EDF binary file → 400.
- NaN input: Verify graceful error response instead of crash.
- Oversized file: File exceeding 100 MB → 413.
- Path traversal: /api/results/../../../etc/passwd → 403.

5.5  System Testing

Full Pipeline Tests (tests/system/):
- test_determinism.py: Run inference on same input twice with identical
  random seed, verify identical output (bitwise reproducibility).
- test_memory.py: Profile memory usage during inference and CMA-ES on
  multiple sequential samples; verify no leaks exceeding 500 MB baseline.
- test_throughput.py: Measure inference throughput (samples/sec) for
  batch sizes 1, 8, 16; verify GPU utilization.
- test_device_fallback.py: Verify model loads and runs correctly on CPU
  when CUDA is unavailable or explicitly forced.

5.6  Non-Functional Testing

5.6.1  Performance and Scalability

- Inference Latency: Measured at 5–10 ms per 2-second EEG window on GPU
  (NVIDIA RTX 3080), 50–100 ms on CPU (8 cores). Within the 60-second
  timeout enforced by the API.
- CMA-ES Convergence: Population=14, max 30 generations. Typical convergence
  within 20–25 generations (~5–10 minutes on CPU). Monitored via WebSocket
  progress stream.
- Synthetic Data Generation: 23,000 simulations with 5 windows each,
  parallelized over 16 CPU cores via joblib. Throughput ~500 simulations/hour.
- Frontend Load: 3D brain rendering (Plotly Mesh3d) tested on Chrome and
  Firefox; renders within 1 second for the fsaverage5 mesh (~20,000 vertices).
- Concurrent Requests: Backend tested with 5 concurrent /api/analyze requests;
  job tracker serializes CMA-ES per request (thread lock on job dict).
  Concurrent biomarker-mode requests process in parallel without contention.

5.6.2  Security

- CORS: Backend enforces explicit allowed origins, methods (GET, POST,
  OPTIONS), and headers (Content-Type, Authorization).
- Rate Limiting: 100 requests/minute per IP for standard endpoints.
- Input Validation: Pydantic models validate all request fields; file uploads
  checked for extension, MIME type, and size before processing.
- Session Security: WebSocket connections validated with job-specific tokens.
- Path Traversal: Results endpoint resolves paths relative to sandbox
  directory, rejecting parent-directory traversals.

5.6.3  Robustness and Error Handling

- Model Unavailable: /api/health returns model_loaded=false; /api/analyze
  returns 503.
- CMA-ES Non-Convergence: Returns best result after max_generations with
  convergence warning, not an error.
- WebSocket Disconnect: Frontend reconnects with exponential backoff
  (1, 2, 4, 8 seconds, max 5 retries).
- Disk Full: Backend monitors output directory usage; rejects new jobs if
  disk usage exceeds 90% of available space.
- CUDA OOM: Falls back to CPU inference with log warning; CMA-ES always
  runs on CPU.

5.7  Test Execution and Reporting

5.7.1  Execution Strategy

- Automated: 149 tests configured in pytest with markers (unit, functional,
  integration, system, regression, slow). CI/CD via GitHub Actions on push
  and pull request, excluding system and integration tests (which require
  model checkpoint and GPU). Full test suite runs locally with:
    pytest tests/ -m "not slow" -v
- Manual: End-to-end test script (scripts/test_e2e.sh) validates health
  check → synthetic sample → EDF upload (both modes). Smoke test
  (scripts/smoke_test.sh) provides quick validation of core functionality.
- Coverage: Measured with pytest-cov; target ≥70% line coverage on src/
  modules.

5.7.2  Reporting and Issue Tracking

- Test results captured in terminal output with pytest -v.
- CI failures reported via GitHub Actions status checks on pull requests.
- Issues tracked in repository; regression tests added for each bug fix.

5.7.3  Conclusion

All 149 tests pass on the current checkpoint (v2.0-submission). The test suite
provides comprehensive coverage of model correctness, API functionality, error
handling, and pipeline integration. On the synthetic3 test dataset (10,000 windows),
the PhysDeepSIF network achieves:

6  Deployment and System Integration

6.1  Deployment Details

The system is containerized for deployment via Docker Compose:

Backend Container (deploy/Dockerfile.backend):
- Base: python:3.9-slim
- Dependencies: PyTorch 2.1, FastAPI, uvicorn, MNE, TVB, h5py, cmaes, scipy,
  numpy, plotly, nibabel, pydantic.
- Data: Required files (checkpoint_best.pt, normalization_stats.json,
  leadfield, connectivity, region labels, test dataset) mounted as volumes.
- Command: uvicorn backend.server:app --host 0.0.0.0 --port 8000 --workers 4
- Healthcheck: curl /api/health every 30s, 10s timeout, 3 retries.

Frontend Container (deploy/Dockerfile.frontend):
- Base: node:20-alpine
- Build: npm install --legacy-peer-deps && npm run build
- Command: npm start (Next.js production server on port 3000)
- Environment: PHYSDEEPSIF_BACKEND_URL set to backend service hostname.

Orchestration (deploy/docker-compose.yml):
- Two services on bridge network 'physdeepsif-net'.
- Frontend depends_on backend with condition: service_healthy.
- Restart policy: unless-stopped.
- Volume mounts: ./data:/app/data, ./outputs:/app/outputs.

Production manual deployment:
- Backend: uvicorn backend.server:app --host 0.0.0.0 --port 8000 --workers 4
- Frontend: cd frontend && npm run build && npm start
- Recommended: nginx reverse proxy with Let's Encrypt for HTTPS termination.

System requirements for deployment:
- CPU: 8+ cores (16 recommended for CMA-ES parallelism).
- GPU: NVIDIA with CUDA 11.8+ and ≥8 GB VRAM (optional; CPU fallback works).
- RAM: 16 GB minimum, 32 GB recommended.
- Storage: ≥15 GB free (datasets ~13 GB, model ~5 MB, outputs variable).

6.2  System Integration Details

The system integrates four major components through well-defined interfaces:

1. Synthetic Data Pipeline → Model Training:
   - HDF5 datasets with standardized schema (eeg, source_activity,
     epileptogenic_mask, x0_vector) serve as the interface.
   - Training script reads from data/synthetic3/ directory; output is
     checkpoint_best.pt + normalization_stats.json.
   - Paths configurable via config.yaml and CLI arguments.

2. Model Training → Inference API:
   - Backend loads checkpoint_best.pt at startup via the model loading
     module (physdeepsif.py).
   - Normalization statistics loaded from JSON for input preprocessing.
   - Inference function wrapped in asyncio.wait_for(60s) for timeout safety.

3. Inference API → Frontend Dashboard:
   - REST API (/api/analyze, /api/results, /api/health, /api/test-samples).
   - WebSocket (/ws/progress/{job_id}) for real-time CMA-ES status.
   - Results served as HTML (Plotly brain heatmaps), JSON (EI scores,
     metrics), and CSV (region data tables).
   - Frontend fetches results via proxy configuration (next.config.mjs
     rewrites) or direct URL.

4. CMA-ES Optimizer → Concordance Engine:
   - Fitted x₀ vector (76 float values) passed to sigmoid → biophysical EI.
   - Biophysical EI top-10 compared with Phase A top-10 for overlap count.
   - Concordance tier (HIGH/MODERATE/LOW) returned to frontend with overlap
     region names.

Integration validation:
- End-to-end test script (scripts/test_e2e.sh) validates all integration
  points: backend health → synthetic sample analysis → EDF upload
  (biomarkers + source_localization modes).
- CI/CD pipeline (GitHub Actions) runs tests on each push, excluding tests
  requiring GPU or full model loading.

The system can also be started locally using the orchestrator script:
  ./start.sh [--backend|--frontend|--check|--kill]
This script validates dependencies, required files, Python/Node.js versions,
and SWC binary integrity before launching services.
